WebBuilder — Estado actual y problema pendiente
Qué es el proyecto:
WebBuilder es una app Django que genera proyectos Django completos e independientes a partir de una URL de API pública y un prompt del usuario. El usuario introduce una URL (ej: https://fakestoreapi.com/products), describe el sitio que quiere, y el sistema genera un proyecto Django completo con models, views, templates, management command de carga de datos, Dockerfile, etc.
Dónde nos hemos quedado:
El sistema está construido y funcionando estructuralmente. El flujo completo es:

Usuario introduce URL + prompt → análisis del dataset → generación del schema (planner) → edición del schema → generación del proyecto completo → descarga ZIP

Los archivos clave del nuevo sistema son:

WebBuilder/utils/llm/generator_prompts.py — prompts para cada archivo que genera el LLM
WebBuilder/utils/generator/project_generator.py — orquestador que llama al LLM 8 veces y ensambla el proyecto
WebBuilder/views/site.py — vistas de preview, generación y descarga ZIP
WebBuilder/utils/llm/client.py — cliente LLM adaptado para OpenRouter y Google

El problema:
El LLM hace 8 llamadas seguidas (pages_structure, models, views, base.html, 3 templates, load_data) y todos los modelos gratuitos dan error 429 por rate limit. Se probó con OpenRouter (qwen/qwen3-coder:free, meta-llama/llama-3.3-70b-instruct:free) y con Google AI Studio (gemini-2.0-flash, gemini-2.0-flash-lite) — todos fallan con limit: 0 o rate limit por minuto.
La solución decidida para mañana:
Meter $10 en OpenRouter (https://openrouter.ai), lo que sube el límite de 50 a 1000 requests/día. El .env quedará así:
LLM_BASE_URL=https://openrouter.ai/api/v1
LLM_API_KEY=sk-or-v1-...
LLM_MODEL=meta-llama/llama-3.3-70b-instruct:free
Y en project_generator.py hay un time.sleep(30) en _llm_call para respetar el límite de 8 requests/minuto.
